# Known limitations

No listed item is classified as blocking for internal release-candidate review. A change in deployment
shape, threat boundary, or user population requires reassessment.

| ID | Description and affected area | User / security / operational impact | Workaround | Severity | Blocking | Future treatment |
| --- | --- | --- | --- | --- | --- | --- |
| KL-001 | Production is a single-host, single-worker SQLite deployment; clustered workers and network filesystems are unsupported. | Users may see reduced throughput; using an unsupported topology risks lock contention and inconsistent operations. | Keep web, worker, database, and artifact roots on one trusted host and use one supervised worker. | Medium | No | Revisit only with a separately designed datastore and lease model. |
| KL-002 | Backup requires web and worker writers to be stopped. | A short maintenance window is required; skipping it invalidates the supported recovery procedure. | Schedule downtime, reconcile, stop writers, pass the explicit confirmation flag, and verify the manifest. | Medium | No | Preserve the fail-closed offline contract unless a future snapshot design proves equivalent consistency. |
| KL-003 | Restore accepts the exact supported migration revision and publishes only into new empty paths. | Operators cannot overwrite in place or automatically downgrade. This reduces corruption risk but increases recovery storage needs. | Restore with the application version recorded by the backup, validate, then switch paths under change control. | Low | No | Add cross-version restore policy only alongside explicit migration compatibility evidence. |
| KL-004 | Windows tests that require creating symlinks can skip when the account lacks `SeCreateSymbolicLinkPrivilege`; junction coverage remains active. | No end-user impact. The local run cannot exercise the OS operation when Windows refuses its creation. | Rely on Linux hosted coverage and Windows junction tests, or rerun locally with Developer Mode/appropriate privilege. | Low | No | Continue monitoring both hosted platforms. |
| KL-005 | Accessibility acceptance uses repository tests plus manual semantic, keyboard, focus, responsive, zoom-token, and reduced-motion review; no external assistive-technology lab or added scanner is used. | Some screen-reader/browser combinations and populated dense datasets are not exhaustively certified. No blocking barrier was observed. | Perform human assistive-technology review before expanding beyond the internal audience and report issues against the existing components. | Low | No | Add broader human testing when audience or UI complexity grows. |
| KL-006 | Phase 29 browser acceptance uses local fixtures and empty review collections; public load tests and customer crawls are prohibited. | Very large real-world result tables and long-running public network behavior are supported by limits/tests but not stress-certified in this phase. | Start with conservative configured limits, monitor resource use, and raise limits only after controlled local profiling. | Low | No | Add bounded synthetic scale fixtures when a concrete capacity target is approved. |
| KL-007 | TLS termination, certificate renewal, log shipping, off-host backup replication, and supervisor selection remain deployment-owner responsibilities. | Misconfigured surrounding infrastructure can reduce availability or trust even though the application fails closed on its own configuration. | Follow deployment/preflight documentation and use organization-approved private ingress, monitoring, encrypted backup storage, and supervision. | Medium | No | Integrate with the selected internal platform only after it is named and authorized. |
| KL-008 | Phase 29 does not run or publish a real hosted release candidate, tag, release, or deployment. | Hosted CI validates the published baseline, while the Phase 29 worktree is validated locally until later publication authorization. | After human approval, commit/push separately, require hosted CI, then run the manual candidate workflow at the exact approved commit. | Informational | No | Close during the authorized publication/release process. |
| KL-009 | Sitemap Audit, Link Audit, Internal Links, Images and Alt Text, Structured Data, and Migration QA still require a technical run ID during creation. Metadata Audit now provides the reusable completed-run selector pattern, but broad specialist conversion is outside Phase 29. | Operators must locate and copy an internal identifier from History for these six specialist workflows; this is usable but unnecessarily technical and error-prone. | Select the completed crawl in History, copy its run ID, and confirm the seed URL before submission. | Medium | No | Treat a shared eligible-run selector as existing specialist-module debt and a CSA-01 integration contract requirement before Combined Site Audit implementation. |
| KL-010 | CSA-02 provides deterministic sample-URL rule testing but not preview against retained crawl evidence. The operator-friendly selector and aggregate context belong to later CSA phases. | Operators can verify hypothetical URLs but cannot estimate a rule across a retained run from the settings screen. Accepting a pasted internal ID would violate the approved interaction contract. | Test up to 100 representative URLs and review resolved sources/conflicts. The private preview endpoint reports an explicit deferred state. | Low | No | Implement bounded retained-evidence preview with the shared selector during CSA-03/CSA-05 integration. |
